# RAG 工作流拆解

这份文档对应项目里的真实执行流程。

## 1. 索引阶段

索引阶段的目标，是把原始资料变成“可检索”的结构。

### 读取资料

在这个项目里，资料有三种来源：

- 内置 seed docs
- 用户直接提交的一段文本
- 用户上传的文件

对应代码在 `app/loaders.py`。

### 切块

原始文档通常太长，不能直接整篇拿去做检索。于是我们会把文档切成多个 chunk。

这里用的是 `RecursiveCharacterTextSplitter`。它会尽量按比较自然的边界切，比如空行、换行、中文标点、空格。这样得到的 chunk 更适合后面做 embedding 和检索。

### Embedding

每个 chunk 会被转换成一个向量。这个向量不是给人读的，而是给机器算“语义相似度”的。

在这个项目里，embedding 模型默认是 `text-embedding-3-small`。

### 写入向量库

生成好 embedding 后，把 chunk 和 metadata 一起写入 Chroma。

metadata 里会保留：

- source
- title
- page
- chunk_index

这些信息后面会被用来返回 sources。

## 2. 问答阶段

问答阶段的目标，是先找到相关资料，再让模型生成答案。

### 接收问题

FastAPI 的 `/api/ask` 接口接收一个问题和可选的 `top_k`。

### 检索

服务会把问题拿去向量库里做相似度检索，取回最相关的若干 chunk。

### 组装 context

检索到的 chunk 不会直接原样给用户，而是先拼成一个 context，和用户问题一起交给模型。

### 生成答案

最后让聊天模型基于问题和 context 生成回答。

这一步的关键不是“让模型自由发挥”，而是通过 prompt 明确要求它优先依据已检索到的材料回答。

## 3. RAG 为什么经常调不准

常见原因有：

- chunk 太大，一段里塞了太多主题
- chunk 太小，上下文不够
- top_k 太少，没拿到足够资料
- 文档源本身不完整
- prompt 没有限制模型胡编

所以做 RAG，不只是写几行调用代码，更重要的是学会观察每一步的输入输出。
